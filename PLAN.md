## 0. What ERPNext Already Provides (Reusable)

ERPNext v16 already has these, which fit Vietnam directly:

| ERPNext Feature | Vietnam Fit? | Notes |
|-----------------|-------------|-------|
| Sales Invoice doctype | Yes, fully | Base for Hóa đơn GTGT / Hóa đơn bán hàng |
| Purchase Invoice doctype | Yes, fully | Hóa đơn đầu vào |
| Customer / Supplier with tax info | Yes | Add MST field to existing Company/Party |
| Print Format (Jinja + HTML) | Yes | Hóa đơn mẫu số 01GTKT0/001... |
| Multi-company in one site | Yes | 1 Frappe "Company" = 1 tenant's legal entity |
| Multi-site (separate DB) | Yes, for max isolation | Each client gets their own site |
| Invoice numbering (naming series) | Yes | Ký hiệu + số hóa đơn |
| Address (country, province, city) | Yes, add VN province list | Nơi xuất, nơi đến |
| Tax calculation engine | Yes | Thuế GTGT 0%, 5%, 8%, 10% |
| Dashboard + report system | Yes | Báo cáo tình hình sử dụng hóa đơn |
| Frappe cron/scheduler | Yes | Auto-retry failed GDT submissions |
| Webhook / API framework | Yes | Receive callbacks from GDT |

**What does NOT exist and MUST be built:**
- GDT portal API integration (khác hoàn toàn với NIC India portal)
- Vietnam XML e-invoice schema generator (Circular 78 XML format)
- Digital signature (ký số) integration (USB Token, HSM, or remote signing service)
- QR code for tra cứu hóa đơn on GDT portal
- Mẫu số hóa đơn (form number) + Ký hiệu (series) management per company
- Hóa đơn điều chỉnh / thay thế workflow
- Vietnam tax report: Bảng kê hóa đơn, Báo cáo tình hình sử dụng HĐĐT
- Bảng mã CQT (tax authority codes) for 63 provinces

## 1. India → Vietnam Concept Mapping

| India Concept | India Implementation | Vietnam Equivalent | Vietnam Implementation |
|--------------|---------------------|-------------------|----------------------|
| GSTIN (15-char) | `GSTIN` doctype, validated pattern | MST (Mã số thuế 10/13 digits) | `MST` doctype, simpler validation |
| GST Category | B2B/B2C/SEZ/EXP/... | Phân loại HĐ: Có mã / Không mã CQT | `InvoiceType` select field |
| IRN (Invoice Reference Number) | Generated by NIC portal | Mã tra cứu (from GDT) | `TransactionID` from GDT |
| e-Invoice Log | Tracks IRN generation state | `EInvoiceLog` | Same pattern, VN-specific statuses |
| e-Waybill | Goods movement tracking (India-specific) | ❌ Not applicable to Vietnam | **DELETE entirely** |
| GST Settings | Single `GST Settings` doctype | Tax Department Settings | `VATSettings` / `GDTConfig` doctype |
| HSN Code | 4/6/8 digit commodity code | ❌ Not required in VN | DELETE (or repurpose for VN commodity code if needed) |
| ITC (Input Tax Credit) | Complex India ITC classification | Khấu trừ thuế GTGT đầu vào | Simplified (just flag: được khấu trừ / không) |
| GST Returns (GSTR-1, GSTR-3B) | Complex periodic filings | Báo cáo tình hình sử dụng HĐĐT | New simple report |
| Reverse Charge (RCM) | India-specific | ❌ Not applicable | DELETE |
| TDS/TCS | Tax deduction at source | ❌ Not applicable | DELETE |
| Purchase Reconciliation | 2A/2B matching | Not needed (no counterpart in VN) | Optional: reconcile input invoices with GDT |
| Ineligible ITC | India-specific | Không khấu trừ (đơn giản hơn) | Simplified boolean |
| Audit Trail | Required for India | Optional for VN | Keep if useful |
| Sandbox Mode | NIC test portal | GDT test/sandbox environment | Keep pattern, different endpoint |
| E-commerce operator GST | India-specific | ❌ Not applicable | DELETE |

## 2. What ERPNext Fields Already Fit Vietnam

### Sales Invoice fields that work as-is:
- `customer` → Khách hàng (bên mua)
- `company` → Đơn vị bán
- `posting_date` / `due_date` → Ngày lập / ngày đến hạn
- `items` → Bảng kê hàng hóa/dịch vụ
- `taxes` → Thuế GTGT 0%, 5%, 8%, 10%
- `total`, `grand_total`, `rounded_total` → Tổng tiền
- `currency` → VND
- `terms` → Điều khoản thanh toán
- `naming_series` → Ký hiệu mẫu số + series

### Party (Customer/Supplier/Company) fields that work:
- `customer_name` / `company_name` → Tên đơn vị
- `address` → Địa chỉ
- `tax_id` (existing ERPNext field!) → MST (Mã số thuế)
- `email` / `phone` → Email / SĐT (required for GDT)
- `bank_account` → Tài khoản ngân hàng

## 3. Implementation Phases

### Phase 1: Scaffold & Clean (Week 1-2)

1. Initialize new Frappe app:
   ```
   bench new-app vietnam_compliance
   ```
2. Copy over structural framework from India Compliance:
   - `hooks.py` pattern (simplified - remove India doctypes from overrides)
   - `install.py` / `uninstall.py` pattern
   - `setup/` pattern for custom fields & property setters
   - `boot.py` for client-side boot info
3. Remove all India-only modules (see DELETE list above)
4. Rename remaining modules (see REWRITE list above)
5. Set up `constants/` with Vietnam data:
   - `PROVINCES` (63 provinces + codes)
   - `VAT_RATES` (0%, 5%, 8%, 10%)
   - `INVOICE_TYPES`
   - `INVOICE_FORM_TEMPLATES` (Mẫu số 01GTKT0/001, 02GTTT0/001...)

### Phase 2: Core Doctypes (Week 2-4)

1. **Tax Department Config** (`TaxDepartmentConfig`) - replaces India's `GST Settings`:
   - GDT API endpoint (production/sandbox)
   - Username/password per company
   - Enable/disable e-invoice
   - Auto-generate on submit
   - Auto-cancel workflow
   - Retry configuration

2. **MST** (`MST`) - replaces India's `GSTIN`:
   - 10 or 13 digit tax code
   - Company link (one company = one MST for e-invoice)
   - GDT authentication credentials per MST

3. **E-Invoice Log** (`EInvoiceLog`) - same pattern as India:
   - Reference to Sales Invoice
   - Transaction ID from GDT
   - Submission status
   - Request/response JSON
   - Retry count

4. **Custom Fields** on `Sales Invoice` - add Vietnam e-invoice fields:
   - `einvoice_status`, `transaction_id`, `form_number`, `series`, `adjustment_type`, etc.

5. **Custom Fields** on `Company` - add MST, digital cert reference, GDT credentials

6. **Property Setters** - make VN fields visible in UI

### Phase 3: GDT API Integration (Week 4-6)

1. **GDT E-Invoice API Client** (`api_classes/gdt/`):
   - Authentication (likely username/password + digital certificate)
   - `submit_invoice()` - Gửi hóa đơn
   - `get_invoice_status()` - Tra cứu trạng thái
   - `cancel_invoice()` - Hủy hóa đơn
   - `adjust_invoice()` - Điều chỉnh hóa đơn
   - `replace_invoice()` - Thay thế hóa đơn
   - Sandbox mode support

> **Research needed**: GDT API documentation. The actual endpoints, auth mechanism, and XML schema per Circular 78/2021/TT-BTC.

2. **XML Generator** (`utils/xml_generator.py`):
   - Map Sales Invoice data to VN e-invoice XML schema
   - Required fields: invoice number, date, buyer/seller info, items, tax, totals
   - Digital signature embedding

3. **Digital Signature Integration**:
   - USB Token signing (via PKCS#11 or Windows CryptoAPI)
   - Remote signing service (HSM-based, common VN providers: VNPT, Viettel, BKAV, MISA...)
   - Sign XML with XAdES

### Phase 4: UI & Workflow (Week 6-8)

1. **Client Scripts** (JS - adapted from India Compliance):
   - `sales_invoice.js` - Add e-invoice actions (Submit to GDT, Cancel, Check Status)
   - `einvoice_status` visual indicator
   - QR code display on invoice

2. **Print Format** (HTML/Jinja):
   - Vietnam e-invoice print template per Circular 78
   - QR code rendering
   - Digital signature watermark
   - Hóa đơn GTGT mẫu 01GTKT0/001

3. **Workspace** (`VAT Vietnam` workspace):
   - Dashboard cards (Invoices pending, Submitted today, Errors)
   - Quick actions (Submit batch, Check batch status)
   - Reports (Tình hình sử dụng HĐĐT, Bảng kê hóa đơn)

### Phase 5: Reports & Scheduler (Week 8-10)

1. **Reports**:
   - Báo cáo tình hình sử dụng hóa đơn điện tử
   - Bảng kê hóa đơn hàng hóa dịch vụ bán ra
   - Bảng kê hóa đơn hàng hóa dịch vụ mua vào
   - Summary dashboard

2. **Scheduler Jobs**:
   - Auto-retry failed submissions (every 5 mins)
   - Auto-generate e-invoice on Sales Invoice submit (if enabled)
   - Periodic status sync with GDT portal
   - Certificate expiry check (cảnh báo chứng thư số sắp hết hạn)

3. **Multi-tenant Setup**:
   - Same pattern as India Compliance's global config
   - `bench set-config -g gdt_api_url <url>`
   - Per-company MST + digital certificate
   - One Frappe site per client company OR multi-company within one site

## 4. Key Architecture Decisions

### Decision 1: Add-on to ERPNext vs Standalone

**Recommendation: Add-on to ERPNext** (same as India Compliance)
- ERPNext already has all prerequisite doctypes (Sales Invoice, Customer, Company, Tax)
- Zero integration overhead - hooks directly into invoice lifecycle
- Reuses ERPNext's multi-tenancy (multi-site, multi-company)
- Faster to market than building standalone from scratch

### Decision 2: One Site per Client vs Multi-Company in One Site

**Recommendation: One site per client company** for SaaS
- Maximum data isolation (each client = separate database)
- Independent backup/restore per client
- Separate GDT credentials per site
- Can use India Compliance's global config pattern for shared infrastructure settings
- ERPNext per-site overhead is ~200MB RAM - manageable for SaaS

### Decision 3: Digital Signature Approach

**Recommendation: Remote signing service integration** (not USB Token)
- USB Token requires physical device per server - bad for SaaS
- VN providers (VNPT SmartSign, Viettel CA, BKAV CA) offer API signing services
- Pattern: submit invoice XML to signing service → get signed XML → submit to GDT
- Abstract signing into a pluggable interface for different providers

### Decision 4: Single vs Dual Invoice System

**Recommendation: Separate `EInvoiceData` child doctype** on Sales Invoice
- Vietnam e-invoice has data not relevant to ERPNext core (form number, series, GDT response...)
- Keep e-invoice metadata in a child table or JSON field, not polluting core Sales Invoice fields
- India Compliance embeds directly (IRN field on Sales Invoice) - we can do cleaner separation

## 5. Risk Areas

| Risk | Mitigation |
|------|-----------|
| GDT API documentation inaccessible or incomplete | Work with GDT or use reverse-engineered APIs from VN developers |
| Digital cert signing complexity | Start with mock/sandbox signing, integrate real signing service later |
| XML schema changes by GDT | Keep schema versioned, watch for Circular updates |
| Multi-tenant performance at scale | Monitor site count per server; prepare for horizontal scaling |
| Legal/regulatory certification required | May need to register as T-VAN (tổ chức cung cấp dịch vụ HĐĐT) |
